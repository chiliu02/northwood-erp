SalesOrderFulfilmentSaga

One row per customer order. The prepayment branch is added at started (snapshotted from payment_terms on saga.data).

Inventory-as-MRP-orchestrator (2026-05-29): the flip — inventory is the single make-vs-buy decision + trigger point. Sales no longer routes shortages to manufacturing. On a partial/failed reservation, inventory raises ReplenishmentRequest(reason=sales_order_shortage) in the same transaction as the reservation (routing manufactured → manufacturing make-to-stock WO / purchased → purchasing itself) and the saga parks at stock_reservation_incomplete. When every short line's inventory.ReplenishmentFulfilled lands the saga re-enters stock_reservation_requested to retry; if any short line's replenishment is cancelled (inventory.ReplenishmentCancelled — unsourceable SKU / no active BOM / no approved vendor) the saga goes to rejected. The manufacturing_requested / manufacturing_in_progress / manufacturing_completed / purchasing_requested states and the make-to-order leg (and events sales.ManufacturingRequested, manufacturing.ManufacturingDispatched, sales.SalesOrderPurchasingRequested) were retired in that flip.

WorkOrderSaga

One row per work order — now purely make-to-stock (the sales-driven started entry was retired). Every WO is seeded directly at work_order_created by WorkOrderReleaseService.releaseForReplenishment — the root replenishment WO and each sub-assembly child; raw-material shortage parks until a covering goods-receipt arrives.

PurchaseToPaySaga

One row per purchase order. No compensation flow today. Carries a nullable sales_order_header_id — non-null when the PO originates from a sales-order-driven replenishment (sales_order_shortage or order_pegged), threaded in via ReplenishmentRequested.sourceSalesOrderHeaderId → convertFromRequisition → started(poId, salesOrderHeaderId); every milestone span stamps northwood.sales_order_id so a Tempo trace from a sales order resolves its buy-side PO sub-saga (manual / reorder-point PRs carry null).

ReplenishmentRequest (aggregate, not a Saga)

One row per (product, warehouse) replenishment cycle — partial unique index enforces one-open-per-SKU/warehouse for reorder_point_breach + work_order_shortage reasons; the per-sales-order-line reasons sales_order_shortage and order_pegged are excluded from the index so multiple SO-driven requests for the same SKU can stack (each back-referenced to a sales-order line via source_sales_order_line_id). Raised by the in-tx reorder-point detector on every on-hand decrement (shipment / stock-adjustment), by the inbox bridge handler on manufacturing.RawMaterialShortageDetected, and in-tx by StockReservationService.reserveForSalesOrder on a partial/failed sales-order reservation (sales_order_shortage) or a to_order line (order_pegged, full line qty, free-stock reservation skipped). target_service (manufacturing | purchasing) is decided in inventory by the SKU's product_card.is_purchased / is_manufactured snapshot — manufacturing and purchasing don't decide where work routes, they only receive it, and signal back ReplenishmentUndispatchable (no BOM / no vendor) so inventory cancels.

reason (reorder_point_breach | work_order_shortage | sales_order_shortage | order_pegged) carries why the request was raised; dispatched_aggregate_kind + dispatched_aggregate_id identify the downstream WO / PR; linked_purchase_order_id stamps the PO that ends the purchasing leg so a later GoodsReceived can resolve back to the request. Emits inventory.ReplenishmentRequested on the request transition, inventory.ReplenishmentFulfilled on markFulfilled, and inventory.ReplenishmentCancelled on markCancelled — the failure counterpart, which for a sales_order_shortage / order_pegged request flips the originating sales order to rejected. ReplenishmentFulfilled.pegged is set = (reason == order_pegged): an order-pegged request's output is reserved to the originating SO line atomically with the stock credit (WO completion or goods receipt), so pegged=true tells sales to ship direct without a re-reservation retry.
