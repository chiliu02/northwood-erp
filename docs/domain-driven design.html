Northwood ERP — Domain-Driven Design

Strategic and tactical DDD analysis of the Northwood codebase: a multi-context, event-driven ERP showcase built on Spring Boot 4 / Spring Data JDBC / Kafka. Curated against Evans' patterns — bounded contexts, aggregates, value objects, domain events, repositories, and process managers — with the project's load-bearing invariants called out where they sharpen the textbook pattern.

1 · Domain overview

Northwood models a discrete-manufacturing ERP — the lifecycle of a physical product from sales quote through fulfilment (stock reservation, make-to-order manufacturing, shipment) to cash collection — alongside its inbound counterpart: procurement of raw materials, goods receipt, three-way match, and supplier payment. Seven services collaborate by publishing domain events; no service calls another's REST endpoints.

The codebase is intentionally a showcase rather than production: it exists to be read at least as much as it is run. Most architectural decisions (events-as-navigation-anchor, schema-per-service-in-one-database, Spring Data JDBC over JPA, ports-and-adapters with a strict 4-way layering ban) optimise for readers tracing data flow with nothing but an IDE.

3 · Strategic design — bounded contexts

Seven bounded contexts, each a separate Maven module, each owning a private PostgreSQL schema authenticated by a dedicated database role. Cross-schema joins are physically blocked by the per-service search_path — the schema-per-service-in-one-database arrangement is a deployment shortcut for the showcase, not a coupling. The database-per-service migration is effectively a config change to application.yml.

The only inter-service contract is the outbox. No "quick REST endpoint" between services. Every cross-context interaction goes through an event.

Aggregate-root stamping — the three-criterion test

Events identify the aggregate the fact is about, not the module that emitted them. A module-A-emits-event-B-about-aggregate-C-owned-by-module-D arrangement is legitimate only when all three hold:

  • Subject match — the named aggregate is genuinely the subject of the fact.
  • Source authority — the emitter is the natural source of knowledge; no other module can synthesise the value from its own state.
  • No invariant claim — the emitter isn't claiming jurisdiction over the named aggregate's lifecycle.

This is orthodox DDD properly read. A happy consequence: every event keyed to a saga's correlation aggregate lands on the same Kafka partition (consumed in order, no cross-partition joins).

Events jars — the Published Language

Each producing service ships a sibling <service>-events Maven module containing every event record it emits. Consumers depend on the events jar, not on the producing service. The jar has only:

  • Domain-event records implementing DomainEvent
  • VOs that events transitively reference (e.g. ApprovedVendor)
  • shared-kernel as its only Maven dep — no Spring, no JDBC, no aggregates

5 · Layering (Hexagonal 4-way rule)

api/
REST controllers, request / response DTOs. Imports only application/.
application/
@Service @Transactional use cases, DTOs (*View / *Command), saga managers, inbox handlers. Imports domain/ and shared-kernel.
domain/
Aggregates, value objects, domain events, repository interfaces. Imports shared-kernel only. No Spring, no JDBC, no Jackson.
infrastructure/
Jdbc*Repository, Jdbc*QueryPort, Kafka publisher, saga worker. Imports application/ + domain/; not api/.
shared-kernel
Framework-free shared types (Money, Quantity, Sku, DomainEvent, SagaInstance).
LayerMay import fromMay NOT import from
api/ application/ only domain/, infrastructure/
application/ domain/, other application/, shared-kernel, shared.applicationapi/, infrastructure/
domain/ shared-kernel only Everything else
infrastructure/ application/, domain/, shared-kernel api/ (one documented @AutoConfiguration exception)
Why api/ → domain/ is banned. A controller holding an aggregate can silently invoke a domain mutator (product.discontinue()) without going through the application service that drains pendingEvents to the outbox. Domain alone is not an integrity boundary; the application service is. The strict ban removes the footgun structurally — a controller that has only a ProductView in scope literally cannot invoke a domain mutator, regardless of intent.
The JdbcTemplate ban. Every JDBC call in *.application.* is a code-review fail. Persistence is one layer down. The check is a one-line grep: Grep 'JdbcTemplate' **/application/**/*.java → zero matches.

6 · Aggregates & the aggregate template

Each aggregate has one root entity, an identity VO, a private constructor, two static factories, and a list of pending events drained at save() time. The same shape repeats across the codebase so a reader who has read Product can find the same anatomy in SalesOrder, WorkOrder, PurchaseOrder, SupplierInvoice, Payment, … 

public final class Product {
    public static final String AGGREGATE_TYPE = "Product";

    // identity + state
    private final ProductId id;
    private final Sku sku;
    private String name;
    private Money salesPrice;
    private Status status;
    private long version;
    private final List<DomainEvent> pendingEvents = new ArrayList<>();

    private Product(...) { ... }

    // creation — emits ProductRegistered
    public static Product register(ProductId id, Sku sku, String name, Money salesPrice, ...) {
        Objects.requireNonNull(id, "id");
        Objects.requireNonNull(sku, "sku");
        if (name.isBlank()) throw new IllegalArgumentException("name must not be blank");
        Product p = new Product(id, sku, name, salesPrice, Status.ACTIVE, /* version */ 0);
        p.pendingEvents.add(new ProductRegistered(...));
        return p;
    }

    // reconstitution from DB — emits nothing
    public static Product reconstitute(...) { ... }

    // intent-named mutator — guards then mutates then emits
    public void changeSalesPrice(Money newPrice) {
        if (status != Status.ACTIVE) throw new IllegalStateException("...");
        if (newPrice.equals(this.salesPrice)) return;          // no-op suppression
        this.salesPrice = newPrice;
        this.pendingEvents.add(new ProductSalesPriceChanged(...));
    }

    public List<DomainEvent> pullPendingEvents() {
        List<DomainEvent> events = List.copyOf(pendingEvents);
        pendingEvents.clear();
        return events;
    }
}

Aggregate-template invariants

  • Identity by VO, not raw UUID — ProductId, SalesOrderId, WorkOrderId, … one per aggregate.
  • Private constructor + two static factories — register(...) for creation (emits *Registered / *Created), reconstitute(...) for hydration from the repository (emits nothing).
  • Intent-named mutators — changeSalesPrice, discontinue, placeOrder, cancel, recordShipped. Never setters.
  • Optimistic concurrency via version BIGINT — UPDATE … WHERE id = ? AND version = ?; zero rows affected ⇒ OptimisticLockingFailureException.
  • Insert vs update by version == 0 — newly minted aggregates are version 0; reconstituted ones carry whatever the DB returned.
  • Pending events drained in the same transaction — repository.save() calls aggregate.pullPendingEvents() and writes each to the outbox table inside the application service's @Transactional. Atomic transactional outbox.
  • Cross-aggregate references by ID only — no object references between aggregates, even within one schema. A SalesOrderLine carries productSku, not Product.
  • Aggregate-type and event-type as constants — Product.AGGREGATE_TYPE + ProductRegistered.EVENT_TYPE; no raw string literals anywhere.

Known exception — BOM is not yet a proper aggregate. Manufacturing exposes a BomEditRepository port that uses the *Repository suffix without a backing aggregate root — today its invariants (acyclic graph, single-active-per-product, no-edit-on-active) live one layer up in BomEditService + BomCycleDetector + the DB partial unique index uq_bom_active_per_product. This violates the convention captured in docs/conventions.md — no *Repository without an aggregate. Promotion to a real Bom aggregate root (with intent-named mutators addLine / removeLine / activate, pendingEvents draining at save(), and BomCycleDetector passed into activate as a domain service) is tracked in dev-todo.md §2.16. New code must not introduce further offenders.
Aggregate guard test rule. Every aggregate has a sibling <AggregateName>Test.java: plain JUnit + AssertJ, no Spring. The test covers null-rejection on every requireNonNull, blank-rejection on every isBlank check, status-guard rejections on every state-machine guard, no-op suppression where the aggregate skips emission, and happy-path event emission. Aggregate guards are load-bearing invariants — a missed requireNonNull lets a null seep through the outbox JSON onto the bus and surfaces hours later as an NPE in a downstream service.

7 · Value objects

Value objects appear in three population layers:

Shared-kernel VOs — cross-context vocabulary

  • Money — amount + currency, arithmetic operations, attribute-based equality. Closure of operations: add, subtract, multiply all return Money.
  • Quantity — numeric + unit, immutable.
  • Sku — the cross-context product identifier carried on every event referencing a product.

Per-aggregate identity VOs

  • ProductId, SalesOrderId, CustomerId, StockItemId, StockReservationId, GoodsReceiptId, ShipmentId, WorkOrderId, RoutingId, SupplierId, PurchaseRequisitionId, PurchaseOrderId, SupplierInvoiceId, CustomerInvoiceId, PaymentId, JournalEntryId.
  • Each wraps a UUID behind an of(UUID) static factory. At the API boundary, services accept raw UUID and wrap on the first line of the method — identity VOs flow through application/ + domain/ only.

Aggregate-local VOs

  • ApprovedVendor (Product) — carried as an immutable list inside Product; transitively referenced by ApprovedVendorListChanged, so it lives in product-events too.
  • RoutingOperation (Manufacturing routing).
  • Status enums (Customer.Status, Product.Status) are nested VOs with STATUS_* string constants for wire-format projection.
Why Primitive Obsession is taken seriously here. Raw UUID at the API boundary is fine. Raw UUID, BigDecimal, or String inside aggregates and events is a code-review fail. The cost of wrapping is one constructor call; the benefit is that mixing up ProductId and SupplierId at a call site becomes a compile error rather than a runtime data-quality incident.

8 · Domain events as load-bearing artifact

Event-driven systems trade direct call graphs for decoupled producers and consumers — which usually means losing "where does this go next?" as a question answerable from source code. Northwood buys that property back by making event classes the load-bearing artifact for cross-service traceability. Every messaging convention in the codebase serves three operational tests:

  1. Find Usages on the event class returns every emitter and every consumer. No FQN inline references, no class-by-string lookups, no dynamic dispatch on string keys.
  2. A consumer's import block is its event-dependency table of contents. Read the imports of any inbox handler / saga manager / projection and you have the complete list of events it reacts to.
  3. Renaming an event or its wire-format identifier breaks at compile time at every dependent site. No string drift; no late-binding surprises from a runtime topic mismatch.

Derived rules

RuleMechanism
Every event type string is hosted as a constant on its event class public static final String EVENT_TYPE = "finance.CustomerInvoiceCreated";
Every aggregate-type string is hosted on the aggregate root SalesOrder.AGGREGATE_TYPE, Product.AGGREGATE_TYPE, …
Cross-service status strings carried on events live on the event class StockReserved.STATUS_RESERVED, CustomerPaymentReceived.INVOICE_STATUS_PAID, …
Distinct Java class per wire-format suffix — never collapse InventorySalesOrderCancellationApplied ≠ ManufacturingSalesOrderCancellationApplied even when payload shape matches.
Inbox-handler registration passes <Event>.class + <Event>.EVENT_TYPE Constructor of AbstractInboxHandler<E>. Structural Java reference, not a string-keyed registry.

Transactional outbox + idempotent inbox

Events go through a transactional outbox: the same @Transactional boundary that mutates the aggregate writes the rows to outbox_message. A scheduled drainer publishes to Kafka. On the consumer side, an inbox table with a per-handler consumer_name dedupes at-least-once redelivery.

9 · Repositories, ports, lookups, projections

The codebase distinguishes five suffixes that all touch persistence, each with a different architectural meaning. They share JDBC as an implementation technology but answer different design questions.

SuffixReturnsMutates?LayerUse case
*RepositoryAggregate rootYesInterface in domain/, impl in infrastructure/persistence/ DDD Repository: collection illusion over the aggregate. Drains pendingEvents to outbox at save().
*QueryPort*View rows / listsNoInterface in application/, impl in infrastructure/persistence/ CQRS read-side: whole-row reads for screens; bypasses the aggregate.
*LookupSingle valueNoInterface in application/, impl in infrastructure/persistence/ Narrow operational lookup: findUnitPrice, findStandardCost. Called by services and inbox handlers alike.
*PortInfrastructure rowsYesInterface in shared.application/, impl in shared.infrastructure/ Inbox / outbox / saga machinery. Not domain.
*ProjectionvoidYes, narrowInterface in application/inbox/, impl in infrastructure/persistence/ Inbox-event-driven read-model writes. Only consumed by *Handler in the same inbox/ package.
Projection vs aggregate — when to promote. Most projection tables (sales.product_pricing, manufacturing.product_active_bom, purchasing.product_approved_vendor, finance.product_accounting, …) are caches of facts owned by another service. Mutation authority lives upstream. Promoting one to a real aggregate would mislead readers — "is this where I change pricing?" No, that's Product.changePricing() in product-service. Today's only legitimate hybrid is inventory.stock_item: projected from product master and a real aggregate with locally-owned reorder policy (StockItem.applyReorderPolicy(...)). Promote a projection only when consumer-side invariants emerge that need a place to live.

Instance-field naming

The type already says what kind of collaborator it is — the field name says what's in it. Use the full aggregate name in plural: salesOrders for SalesOrderRepository, purchaseOrders for PurchaseOrderRepository, journalEntries for JournalEntryRepository, productPricing for ProductPricingLookup. Never repository, repo, lookup, or abbreviated context-implied forms (orders, invoices, pricing) — call sites read like business logic, not data-access plumbing.

10 · Domain & application services

Application services — thin orchestration

Application services are @Service @Transactional use-case orchestrators. They load aggregates, invoke intent-named mutators, save through the repository (which drains pending events to the outbox), and return a *View. They hold no business logic of their own — if logic would naturally live on an aggregate or VO, it goes there. 

@Service
public class ProductService {

    private final ProductRepository products;

    @Transactional
    public ProductView changeSalesPrice(UUID productId, BigDecimal amount, String currency) {
        Product product = products.findById(ProductId.of(productId)).orElseThrow();
        product.changeSalesPrice(Money.of(amount, currency));        // domain logic on aggregate
        products.save(product);                                      // drains pendingEvents to outbox
        return ProductView.from(product);
    }
}

Domain services — rare and named in the ubiquitous language

Used sparingly when an operation doesn't belong to any single aggregate. Real examples in the codebase:

  • CurrencyConverter (finance) — domain port that converts Money across currencies given an exchange-rate source. Single method, throws RateNotFoundException, deliberately stateless.
  • BomCycleDetector (manufacturing) — pure function over a candidate BOM edit + the current active-BOM graph; rejects edits that would introduce a cycle.
  • MaterialsCostRollupService (manufacturing) — multi-input domain operation (vendor prices + active BOM) computing a fact about a Product; emits ProductMaterialsCostComputed stamped aggregate_type = Product via the cross-context emission rule.

The View / Command / Request pattern

All data-shaped records live in application/dto/. Three roles:

  • *View — read-side projection of an aggregate, with a static from(Aggregate) mapper. Controllers return it directly.
  • *Command — input shape for a service command, with Bean Validation (@NotNull, @Valid) annotations. Controllers accept it directly as @RequestBody.
  • api/dto/*Request / *Response — introduced only when wire and application shapes genuinely diverge (e.g. CancelOrderRequest { reason } + path-bound id vs CancelOrderCommand { salesOrderHeaderId, reason }). YAGNI default: no parallel api/dto/ mirror.

11 · Sagas (process managers)

Three long-running cross-context flows are modelled as sagas. Each saga has its own state table in the schema that owns the flow, its own state-machine class, and a dedicated worker + inbox handlers.

SalesOrderFulfilmentSaga sales-service

Drives a placed order from stock reservation, through optional make-to-order manufacturing, to shipment, customer invoice, and full payment.

started → stock_reserved → (stock_reservation_failed | ready_to_ship) → manufacturing_in_progress → manufacturing_completed → goods_shipped → invoice_issued → payment_received (completed) → cancelled : compensating → compensated

Inputs: StockReserved, WorkOrderCreated, WorkOrderManufacturingCompleted, ManufacturingDispatched, ShipmentPosted, CustomerInvoiceCreated, CustomerPaymentReceived, InventorySalesOrderCancellationApplied, ManufacturingSalesOrderCancellationApplied.

Compensation: POST /api/sales-orders/{id}/cancel is rejected with 409 past goods_shipped; otherwise flips header to cancelled and saga to compensating. Inventory + manufacturing each apply idempotently and ack with *.SalesOrderCancellationApplied; when both acks land, saga emits SalesOrderCompensated.

MakeToOrderSaga manufacturing-service

One saga row per work order. Drives BOM-explosion and recursive sub-assembly WO creation, raw-material reservation, and shortage replenishment.

started → work_order_created → raw_material_reservation_requested → (raw_materials_reserved | raw_material_shortage → back to work_order_created) → completed

Sub-assembly recursion: each child WO carries parent_work_order_id and gets its own saga row. Parent-on-children gate: a parent WO whose ops are all done holds at in_progress while any sub-assembly child is unfinished; on the last child, WorkOrder.onChildCompleted(true) cascades up the parent chain in the same transaction.

PurchaseToPaySaga purchasing-service

One saga row per purchase order. Drives PO approval, goods receipt, supplier invoice approval, and payment settlement.

started → purchase_order_approved → waiting_for_goods → goods_received → supplier_invoice_approved → (supplier_payment_made → ) p2p_completed

Auto-approve gated by northwood.purchasing.shortagePoAutoApprove for shortage-driven PRs; manual via POST /api/purchase-orders/{id}/approve. Partial payments park at supplier_payment_made. The three_way_match_* states are reserved for future variance handling.

Saga manager class shape

RoleLocationHolds
Manager interface<svc>.application.saga.<Flow>SagaManager Orchestration verbs only — lifecycle (insertStarted, requestCompensation), drain, one applyXxx per inbox event.
Manager impl<svc>.infrastructure.saga.Jdbc<Flow>SagaManager Extends shared SagaManager<S, P>. Only the typed saga port + ObjectMapper. No outbox, no JdbcTemplate, no other repos.
Worker shell<svc>.infrastructure.saga.<Flow>SagaWorker @Scheduled poll() → manager.drain(BATCH, workerId, this::advance). advance(saga) holds worker-driven side effects.

Two non-obvious rules baked into the saga base — preserve when adding a saga.
  1. Each saga in a batch gets its own transaction (REQUIRES_NEW) so a failed advance rolls back the outbox writes that haven't yet been flushed.
  2. The retry path reloads the saga before saving — the in-memory object may already carry the half-applied mutation that triggered the throw; persisting it on retry would persist the half-state.

12 · Supple design conventions

  • Intention-revealing interfaces — intent-named mutators (discontinue, placeOrder, recordShipped) read as domain operations, not CRUD verbs.
  • Side-effect-free functions on VOs — Money.add, Money.multiply, Quantity.scaleBy return new instances; never mutate in place.
  • Closure of operations — operations on Money return Money; Quantity returns Quantity. Composable arithmetic across aggregates without leaking types.
  • Single return path; exhaustive branches — methods with branching logic converge to one return at the bottom (early-return reserved for true bail-outs). Violated input contracts throw IllegalStateException — never silently coerce.
  • Documented silent fallbacks — any .orElse(SENTINEL) needs Javadoc on emitter (trigger + value + rationale + tightening alternatives), cross-reference on the consumer, a log line when it fires, and a row in docs/design-notes.md. Undocumented fallbacks are code-review fails.
  • Standalone classes — the shared-kernel module imports nothing but the JDK; aggregates import only shared-kernel.
  • Class-member ordering — nested types → static fields → instance fields → constructors → methods (static factories alongside the private constructor). RowMapper lambdas at the bottom of a repository are a code-review fail.

13 · Anti-patterns avoided

Anti-patternHow Northwood avoids it
Anemic domain model Aggregates carry intent-named mutators with guards + event emission; application services are thin pass-throughs.
God aggregate Aggregates kept small; cross-aggregate references by ID only (SalesOrderLine carries productSku, not Product).
Repository for non-roots Repositories only on aggregate roots. Internal entities (SalesOrderLine, PaymentAllocation) loaded through the root.
Leaking infrastructure into domain Greppable invariants: zero JdbcTemplate in application/; domain/ imports only shared-kernel; Jackson lives behind ObjectMapper in infra.
Missing ubiquitous language Aggregates and methods named in business terms (placeOrder, recordShipped, activateBom, discontinue). No *Manager / *Processor / *Helper in the domain layer.
Primitive obsession Per-aggregate identity VOs; Money / Quantity / Sku in shared-kernel.
Broken aggregate invariants Private constructors + static factories + intent-named mutators. No public setters. Optimistic concurrency on version.
No bounded-context boundaries One PostgreSQL schema + role per context; search_path blocks cross-schema joins at runtime; events jars are the only inter-service contract.
Conformist when ACL needed Each consumer service projects upstream events onto a local read model (sales.product_pricing, finance.product_accounting) — the projection acts as a thin Anticorruption Layer between contexts.
Transaction script masquerading as DDD Application services orchestrate; aggregates own the logic + emit events. The grep JdbcTemplate in application/ → zero is the canary.
*Repository without a backing aggregate Convention captured in docs/conventions.md: every *Repository interface in domain/ must have a sibling aggregate root in the same package declaring AGGREGATE_TYPE. Either promote the aggregate or pick a different suffix. One legacy offender (BomEditRepository) is tracked for promotion; new code must not add more.

14 · Evolutionary stance

The model is built to absorb three specific kinds of change cheaply:

  1. Database-per-service split. Per-service connection pools, per-service roles, per-service search_path, per-service Liquibase, no cross-schema FKs. Migration is a change to spring.datasource.url in each application.yml — no code touched.
  2. Event-schema evolution. Jackson tolerance handles additive producer changes silently. Breaking changes (rename, type change) surface as compile errors at every consumer that depends on the producer's events jar — not as runtime deserialisation failures.
  3. Read-model addition. A new screen or report adds a *QueryPort + projection; no aggregate touched. The reporting service has its own master changelog and inbox handlers that subscribe to whatever upstream events it needs.

Areas where evolution is not cheap by design — on purpose:

  • Renaming a domain event is a multi-module change (events jar + every consumer). The compile-time breakage is the safety mechanism.
  • Splitting one bounded context into two is a Hard refactor. The current seven were drawn carefully and are stable.
  • Promoting a projection to an aggregate is a deliberate decision that signals "the consumer context now owns invariants over this data" — it's intentionally a heavyweight change because that ownership claim has cascading consequences.
Northwood as a DDD reference. The model is opinionated — many of the rules (event-class-as-anchor, strict 4-way layering, EVENT_TYPE/AGGREGATE_TYPE constants, full-plural field naming, single-return + exhaustive branches) go beyond textbook DDD to optimise for read-time navigability of an event-driven system. They are best understood as adaptations of Evans' patterns to a codebase meant to be read as much as it is run.
