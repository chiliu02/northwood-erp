Northwood — Domain Event Flow

Service-name suffix -service stripped for compactness. Two views below: the source-first table answers "when this event fires, who hears it and what do they do?" (full consumer + description detail); the destination-first table answers "which events does this service receive, and from whom?" (compact 4-column summary). Cells are merged via rowspan wherever the same value repeats in adjacent rows. Click any value with a ▼ marker to collapse / expand that group — Source Service or Aggregate Root in the first table, Destination Service in the second. Nested collapses are tracked, so collapsing an aggregate and then its source (or vice versa) restores correctly when you expand.

Each row maps a published domain event to a single consumer. Events with multiple consumers occupy multiple rows. The Aggregate Root column reflects the aggregate_type string written to the outbox (and carried on the event envelope). Consumer names come from the CONSUMER_NAME constant on each *Handler in application/inbox/. The Description column names the table(s) and field(s) each consumer writes. Events with no current consumer are listed for completeness.

Grouped by Source Service → Aggregate Root → Domain Event → Destination Service. One row per consumer (so an event with multiple consumers in the same destination service occupies multiple rows).

Notes

  • Inventory's and manufacturing's compensation acks are two physically distinct events — InventorySalesOrderCancellationApplied in inventory-events (aggregate StockReservation) and ManufacturingSalesOrderCancellationApplied in manufacturing-events (aggregate WorkOrder) — each acknowledged by its own sales-side handler. The Java class names are prefixed to avoid a flat-namespace collision; the wire formats (inventory.SalesOrderCancellationApplied / manufacturing.SalesOrderCancellationApplied) deliberately keep the bare event name since the service prefix already conveys the source — see the Javadoc on either event class for the why.
  • Cross-context aggregate-root stamping is intentional, not a pragmatic compromise. The rule: an event names the aggregate the fact is about, not the module that emits it — logical ownership and physical emission are decoupled. A module-A-emits-event-about-aggregate-B arrangement is legitimate when (a) the named aggregate is genuinely the subject of the fact, (b) the emitter is the natural source of authority for that fact, and (c) the emitter isn't claiming jurisdiction over the named aggregate's invariants. Two events in Northwood take this path today: manufacturing.ManufacturingDispatched stamped SalesOrder (manufacturing reports an outcome the SalesOrder needs to advance its saga) and manufacturing.ProductMaterialsCostComputed stamped Product (manufacturing computes a fact about a Product owned upstream). Both pass all three criteria. The Kafka co-partition benefit (every event for a saga's correlation aggregate lands on the same partition) is a happy consequence of correct modeling, not the reason for the choice. Full rule + counter-examples in docs/architecture.md § Aggregate-root stamping and docs/design-notes.md § Cross-context aggregate-root stamping on events.
  • Customer master events (CustomerRegistered, CustomerAddressChanged, CustomerContactChanged, CustomerNameChanged) are published by their aggregates but have no inbox handler yet.
  • StockReserved in sales uses the bare consumer name sales.fulfilment-saga (not sales.fulfilment-saga.stock-reserved) — this is the saga's primary entry point and predates the suffix convention used by later additions.
  • Reporting projections (*_view, production_planning_board, financial_dashboard_daily, material_shortage_view) are all order-tolerant: every write is INSERT ... ON CONFLICT DO UPDATE. Stub rows (e.g. product_sku='(pending)') get backfilled by whichever event arrives first carrying the missing identity.
  • "+=" in the Description column denotes an additive bump (so partial-then-full events sum correctly); plain "→" denotes an overwrite.

Coverage gaps

Four events have no consumer at all, plus a few events whose consumer coverage is incomplete. Listed in priority order — symptom → impact → suggested handler(s). Critical means a downstream service makes business-wrong decisions today; Likely acceptable is documented for completeness.

Likely acceptable — documented for completenessDeferred

1. Customer master events: CustomerRegistered, CustomerNameChanged, CustomerAddressChanged, CustomerContactChanged

Sales owns the customer aggregate and reads it directly for its own checks. Downstream services (finance invoice headers, reporting sales_order_360_view) snapshot customer name + currency from event payloads at order placement time, which is historically correct — overwriting old invoice names after a customer rename would arguably destroy audit value. Only a gap if a UI requirement explicitly says "show current customer name on past orders"; not assumed here.

2. ManufacturingDispatched to production-planning

The production-planning board only gets a row when a work order is actually created. Rejected lines (no BOM, not manufactured) generate ManufacturingDispatched outcomes but no work orders, so there's nothing on the board to update. A "manufacturing rejection rate" dashboard widget could subscribe, but no current read model needs the signal.

Inferred-only — verify against intent before fixingDeferred

  • WorkOrderCancelled has no finance consumer. Today no WIP cost is posted to GL, so nothing needs reversing. Becomes a real gap if WIP capitalisation is added (currently flagged low-priority alongside multi-currency GL consolidation).
  • OperationCompleted is only consumed by reporting. ATP could in principle use partial-operation completion to release planned-quantity progressively, but current ATP waits for full WorkOrderManufacturingCompleted. Not a correctness gap — a precision tradeoff.
  • SalesPriceChanged is only consumed by sales itself. Reporting could keep a price-history projection for trend dashboards, but the demo doesn't require it.
