Northwood — Domain Event Flow

Service-name suffix -service stripped for compactness. Two views below: the source-first table answers "when this event fires, who hears it and what do they do?" (full consumer + description detail); the destination-first table answers "which events does this service receive, and from whom?" (compact 4-column summary). Cells are merged via rowspan wherever the same value repeats in adjacent rows. Click any value with a ▼ marker to collapse / expand that group — Source Service or Aggregate Root in the first table, Destination Service in the second. Nested collapses are tracked, so collapsing an aggregate and then its source (or vice versa) restores correctly when you expand.

Each row maps a published domain event to a single consumer. Events with multiple consumers occupy multiple rows. The Aggregate Root column reflects the aggregate_type string written to the outbox (and carried on the event envelope). Consumer names come from the CONSUMER_NAME constant on each *Handler in application/inbox/. The Description column names the table(s) and field(s) each consumer writes. Events with no current consumer are listed for completeness.

Notes

  • SalesOrderCancellationApplied exists as two physically distinct events — one in inventory-events (aggregate StockReservation), one in manufacturing-events (aggregate WorkOrder) — each acknowledged by its own sales-side handler. They share a class name but are different topics/types.
  • Cross-context aggregate-root stamping is intentional, not a pragmatic compromise. The rule: an event names the aggregate the fact is about, not the module that emits it — logical ownership and physical emission are decoupled. A module-A-emits-event-about-aggregate-B arrangement is legitimate when (a) the named aggregate is genuinely the subject of the fact, (b) the emitter is the natural source of authority for that fact, and (c) the emitter isn't claiming jurisdiction over the named aggregate's invariants. Two events in Northwood take this path today: manufacturing.ManufacturingDispatched stamped SalesOrder (manufacturing reports an outcome the SalesOrder needs to advance its saga) and manufacturing.ProductMaterialsCostComputed stamped Product (manufacturing computes a fact about a Product owned upstream). Both pass all three criteria. The Kafka co-partition benefit (every event for a saga's correlation aggregate lands on the same partition) is a happy consequence of correct modeling, not the reason for the choice. Full rule + counter-examples in docs/architecture.md § Aggregate-root stamping and docs/design-notes.md § Cross-context aggregate-root stamping on events.
  • Customer master events (CustomerRegistered, CustomerAddressChanged, CustomerContactChanged, CustomerNameChanged, CustomerDeactivated), ProductDiscontinued, PurchaseOrderApproved, and ProductMaterialsCostComputed are published by their aggregates but have no inbox handler yet.
  • StockReserved in sales uses the bare consumer name sales.fulfilment-saga (not sales.fulfilment-saga.stock-reserved) — this is the saga's primary entry point and predates the suffix convention used by later additions.
  • Reporting projections (*_view, production_planning_board, financial_dashboard_daily, material_shortage_view) are all order-tolerant: every write is INSERT ... ON CONFLICT DO UPDATE. Stub rows (e.g. product_sku='(pending)') get backfilled by whichever event arrives first carrying the missing identity.
  • "+=" in the Description column denotes an additive bump (so partial-then-full events sum correctly); plain "→" denotes an overwrite.

Coverage gaps

Nine events have no consumer at all, plus a few events whose consumer coverage is incomplete. Listed in priority order — symptom → impact → suggested handler(s). Critical means a downstream service makes business-wrong decisions today; Staleness means a read model drifts but no transaction goes wrong; Likely acceptable is documented for completeness.

Critical — business-incorrect decisions todayCritical

1. ProductDiscontinued has zero consumers

Symptom: a product can be marked discontinued in product-service, but no other service learns of it.

Impact:

  • sales — SalesOrderService.placeOrder checks customer.status but not product status; new orders accept discontinued products.
  • purchasing — PurchaseRequisitionService / PO creation will still procure a discontinued product.
  • manufacturing — ManufacturingRequestedHandler rejects on is_manufactured=false, but ProductDiscontinued doesn't flip that flag, so work orders still spawn for discontinued finished goods. Active BOMs also stay active.
  • reporting — available_to_promise_view and production_planning_board keep showing discontinued items as orderable / plannable.
  • inventory — stock_item keeps firing reorder calculations for a product nobody is buying.

Suggested handlers:

  • sales.product-discontinued → flag the product in sales.product_pricing (or a parallel sales.product_status projection) and reject orders for those products at placeOrder.
  • manufacturing.product-discontinued → set manufacturing.product_replenishment.is_purchased=false, is_manufactured=false so the existing rejection path covers the case; clear manufacturing.product_active_bom for the product.
  • purchasing.product-discontinued → block new POs/PRs.
  • reporting.atp.product-discontinued → stamp discontinued_at on the ATP row so the UI can grey it out.
  • inventory.product-discontinued → suppress reorder alerts on stock_item.

2. ProductCreated has no inventory consumer

Symptom: inventory.StockItemProjection.applyReorderPolicy documents this gap explicitly — "if we have no row for the product yet, the projection silently no-ops … when inventory grows a ProductCreated consumer the row will appear." The Liquibase seed pre-creates stock_item rows for the 5 demo SKUs, which masks the gap during the demo.

Impact: any product registered after boot via POST /api/products cannot have a reorder policy applied — ReorderPolicyChanged projects to a missing row and silently no-ops with a WARN log.

Suggested handler: inventory.product-created → insert an empty inventory.stock_item row (default warehouse MAIN, zero on-hand) at product registration so the later ReorderPolicyChanged has something to update.

3. CustomerDeactivated has no consumers

Symptom: sales already enforces customer.status == ACTIVE locally on order placement (direct DB read of its own customer table), so this isn't a sales-side correctness gap. Downstream services have no signal.

Impact:

  • reporting dashboards continue to count deactivated customers in active-customer totals.
  • finance has no trigger to flag outstanding AR against deactivated customers for collections.

Suggested handlers: a reporting projection that decrements active-customer counts; an optional finance projection that flags AR rows.

Staleness — read models driftStaleness

4. PurchaseOrderApproved has no consumer

Symptom: the event is published, but the purchasing P2P saga worker polls saga rows directly (state-driven) rather than subscribing via inbox. No other service receives the event.

Impact: reporting.purchase_order_tracking_view.po_status is set once by the po-created handler with whatever status the PO has at creation time. When a draft PO is later approved via POST /api/purchase-orders/{id}/approve and the aggregate transitions to 'sent', the tracking view's po_status stays at 'draft' indefinitely. The shortage-driven auto-approve path masks this because approval fires in the same transaction as creation and the create event already carries the post-approval status.

Suggested handler: reporting.po-tracking.po-approved → update purchase_order_tracking_view.po_status='sent' and stamp approved_at.

5. ProductMaterialsCostComputed has no consumer

Symptom: manufacturing rolls up materials cost into manufacturing.product_materials_cost and emits this event, but nothing downstream reacts. Manufacturing's own UI read is the only sink.

Impact: the rollup is a dead-end. When a supplier price drops and triggers a rebased materials_cost, none of:

  • product.product.standard_cost (today maintained by hand via POST /api/products/{id}/standard-cost),
  • finance.product_standard_cost (used by COGS posting at shipment time),
  • reporting.product_standard_cost,

…ever updates. COGS continues to post at the previous standard cost until someone manually re-enters it. This is the cost-chain integration referenced as §2.8 Slice D in the slice docs ("close the cost loop").

Suggested handler: product.materials-cost-rebased (in product-service) → update the product master's standard_cost from the rolled-up value and emit StandardCostChanged, which already has finance + reporting consumers wired.

Likely acceptable — documented for completenessDeferred

6. Customer master events: CustomerRegistered, CustomerNameChanged, CustomerAddressChanged, CustomerContactChanged

Sales owns the customer aggregate and reads it directly for its own checks. Downstream services (finance invoice headers, reporting sales_order_360_view) snapshot customer name + currency from event payloads at order placement time, which is historically correct — overwriting old invoice names after a customer rename would arguably destroy audit value. Only a gap if a UI requirement explicitly says "show current customer name on past orders"; not assumed here.

7. ManufacturingDispatched to production-planning

The production-planning board only gets a row when a work order is actually created. Rejected lines (no BOM, not manufactured) generate ManufacturingDispatched outcomes but no work orders, so there's nothing on the board to update. A "manufacturing rejection rate" dashboard widget could subscribe, but no current read model needs the signal.

Inferred-only — verify against intent before fixingDeferred

  • WorkOrderCancelled has no finance consumer. Today no WIP cost is posted to GL, so nothing needs reversing. Becomes a real gap if WIP capitalisation is added (currently flagged low-priority alongside multi-currency GL consolidation).
  • OperationCompleted is only consumed by reporting. ATP could in principle use partial-operation completion to release planned-quantity progressively, but current ATP waits for full WorkOrderManufacturingCompleted. Not a correctness gap — a precision tradeoff.
  • SalesPriceChanged is only consumed by sales itself. Reporting could keep a price-history projection for trend dashboards, but the demo doesn't require it.
