Composed State Machine — as-built

The sales-order fulfilment state machine as it runs — a parent (order) status that is a brainless fold over its line states (header ship-vocabulary == line vocabulary), with the per-line progress split into independent axes. Described in docs/composed-state-machines.md.

As-built after §2.29 + §2.45 (2026-06-10). A single faithful classify() fold on the aggregate (no blind markStatus writer), the in-progress band carried on the line and mapped 1:1 to the header rung, the invoice/pay axes off the SO line (§12.6), completed as a cross-axis meet, and cancel/amend gates reading line predicates rather than a header allow-list. Closed-gap accounting: composed-state-machines.md §14. The canonical as-built map remains system-map.html.

1 · The order header — a brainless fold plus three terminals

The header carries one SalesOrder.Status, which tracks the shipment axis. Since §2.45 the fold is faithful: the header ship-vocabulary equals the line vocabulary — five fold-region states (open ⊏ partially_reserved ⊏ reserved ⊏ partially_shipped ⊏ shipped, the output of classify(lines), §2) plus three absorbing terminal commands; completed is a cross-axis meet (assert fold == shipped, then invoiced ∧ paid off-line). The lines are the single source of truth; the header is a derived label, so gates read the lines, not the header (the fold is lossy). Cancel is rejected once any goods have shipped — its guard is the line predicate anyLineShipped(), not a status allow-list.

2 · A line is a product of independent axes

Shipping, invoicing, and payment progress independently (you can invoice before shipping, ship before invoicing, or split each at unrelated quantities). A line's full state is therefore the product of one progress lattice per axis — Harel orthogonal regions. A product of lattices is still a lattice, so the fold algebra is untouched (§12). The payment axis lives on finance.CustomerInvoice and is projected back — hence the cross-context note.

3 · Variant coarsening — make / buy / stock onto one band

Different product variants run different internal shipment FSMs, but each coarsens onto the same band via a monotone map gv (§11). The order fold sees only the band, so it never branches on product type. A stock line in reserved, a make line in ready_to_ship, and a buy line in ready_to_ship all coarsen to the reserved band. Since §2.45 the band names are the LineStatus vocabulary (and, equivalently, the header fold region — header ≡ line).

4 · The fold — classify

Line level — quantity buckets → band

band_ship(line) =
    shipped             if shippedQty == orderedQty
    partially_shipped   if 0 < shippedQty < orderedQty
    reserved            if shippedQty == 0
                          ∧ allocatedQty == orderedQty
    partially_reserved  if partial reservation / in production
    open                otherwise
    cancelled           if line soft-removed    (neutral)

band_invoice(line) =
    INVOICED           if invoicedQty == orderedQty
    PARTIALLY_INVOICED if 0 < invoicedQty < orderedQty
    NOT_INVOICED       otherwise

Order level — line multiset → SalesOrder.Status

// over LIVE lines (cancelled filtered):
// meet_a = ⊓ bands_a,  join_a = ⊔ bands_a

classify(order) =
  cancelled           if cancel() OR no live lines   // top-down terminals
  rejected            if reject()
  completed           if complete()  (asserts fold==shipped;
                        invoice∧pay meet off-line — §12.6)
  shipped             if meet == shipped
  partially_shipped   if join ⊒ partially_shipped
  reserved            if meet ⊒ reserved
  partially_reserved  if join ⊒ partially_reserved
  open                otherwise

meet == shipped ⟺ all live lines fully shipped; meet ⊒ reserved ⟺ all live lines reserved or beyond; join ⊒ partially_* ⟺ at least one line has reached that band. Both meet and join are commutative ⇒ the result is order-insensitive (shuffle / replay / batch re-grouping can't change it). One writer: the aggregate's recomputeStatus(), called after every line mutation, with the invariant stored status == classify(lines). The header ship-vocabulary equals the line vocabulary (§2.45); gates read the lines (anyLineShipped() + isTerminal()), not the header — the fold is lossy.

C · Multi-axis: partial ship + non-aligned partial invoice

SO: line 1 = 10×A, line 2 = 10×B (stock). Shipment of 2×A + 4×B; a separate invoice of 5×A + 0×B (deliberately not aligned with the shipment).

lineorderedshipped → band_shipinvoiced → band_invoice
line 1 (A)102 → partially_shipped5 → PARTIALLY_INVOICED
line 2 (B)104 → partially_shipped0 → NOT_INVOICED
 
ship axis    : meet = partially_shipped, join = partially_shipped  →  header.status      = partially_shipped
invoice axis : meet = NOT_INVOICED,      join = PARTIALLY_INVOICED →  header.invoiceFlag = PARTIALLY_INVOICED

Two independent componentwise folds; one status column can't carry both — which is exactly why ship and invoice are orthogonal regions, not one chain. The cross-line bundling (2×A + 4×B in one shipment) is invisible to the fold: each line accumulates only its own Σ, so the result is identical whether the goods arrived in one shipment or several. Boundary: a return would decrease shippedQty — moving an axis backwards and breaking the monotonicity the lattice assumes. Reversals need signed deltas + a net projection, not a status chain (out of scope today).
