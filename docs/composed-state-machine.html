Composed State Machine — ideal / target model

The sales-order fulfilment state machine as it should be — a parent (order) status that is a pure fold over its line states, with the per-line progress split into independent axes. This is the target described in docs/composed-state-machines.md; the live code is a narrower special case.

This is a design target, not the current code. It depicts the model after the dev-todo §2.29 gaps are closed: one classify() fold on the aggregate (no blind markStatus writer), the in-progress band carried on the line, the invoice/pay axes separated, and COMPLETED as a cross-axis meet. For where today's code differs, see composed-state-machines.md §14. The canonical as-built map remains system-map.html.

1 · The order header — a two-tier statechart

The header carries one SalesOrder.Status, which tracks the shipment axis. Four of its states are derived outputs of the line fold (classify, §2); two are command overrides that absorb the machine; one (COMPLETED) is a cross-axis meet requiring all axes saturated. Cancel is rejected once any goods have shipped — so the only edges into CANCELLED leave SUBMITTED / IN_FULFILMENT.

2 · A line is a product of independent axes

Shipping, invoicing, and payment progress independently (you can invoice before shipping, ship before invoicing, or split each at unrelated quantities). A line's full state is therefore the product of one progress lattice per axis — Harel orthogonal regions. A product of lattices is still a lattice, so the fold algebra is untouched (§12). The payment axis lives on finance.CustomerInvoice and is projected back — hence the cross-context note.

      partially_invoiced --> invoiced : remaining units invoiced
    }
    --
    state "Payment axis  (finance.CustomerInvoice, projected)" as P {
      [*] --> not_paid
      not_paid --> partially_paid
      not_paid --> paid
      partially_paid --> paid
    }
  }

3 · Variant coarsening — make / buy / stock onto one band

Different product variants run different internal shipment FSMs, but each coarsens onto the same band via a monotone map gv (§11). The order fold sees only the band, so it never branches on product type. A stock line in reserved, a make line in ready_to_ship, and a buy line in ready_to_ship all coarsen to READY.

4 · The fold — classify

Line level — quantity buckets → band

band_ship(line) =
    SHIPPED            if shippedQty == orderedQty
    PARTIALLY_SHIPPED  if 0 < shippedQty < orderedQty
    READY              if shippedQty == 0
                          ∧ allocatedQty == orderedQty
    IN_PROGRESS        if partial reservation / in production
    NOT_STARTED        otherwise
    CANCELLED          if line soft-removed     (neutral)

band_invoice(line) =
    INVOICED           if invoicedQty == orderedQty
    PARTIALLY_INVOICED if 0 < invoicedQty < orderedQty
    NOT_INVOICED       otherwise

Order level — line multiset → SalesOrder.Status

// over LIVE lines (cancelled filtered):
// meet_a = ⊓ bands_a,  join_a = ⊔ bands_a

classify(order) =
  CANCELLED          if cancel cmd OR no live lines
  REJECTED           if reject cmd
  COMPLETED          if meet_ship    == SHIPPED
                        ∧ meet_invoice == INVOICED
                        ∧ meet_pay     == PAID
  SHIPPED            if meet_ship == SHIPPED
  PARTIALLY_SHIPPED  if join_ship ⊒ PARTIALLY_SHIPPED
  IN_FULFILMENT      if join_ship ⊒ IN_PROGRESS
  SUBMITTED          otherwise

meet_ship == SHIPPED ⟺ all live lines fully shipped; join_ship ⊒ PARTIALLY_SHIPPED ⟺ at least one line shipped ≥ 1 unit. Both meet and join are commutative ⇒ the result is order-insensitive (shuffle / replay / batch re-grouping can't change it). One writer: the aggregate's recomputeStatus(), called after every line mutation, with the invariant stored status == classify(lines).

C · Multi-axis: partial ship + non-aligned partial invoice

SO: line 1 = 10×A, line 2 = 10×B (stock). Shipment of 2×A + 4×B; a separate invoice of 5×A + 0×B (deliberately not aligned with the shipment).

lineorderedshipped → band_shipinvoiced → band_invoice
line 1 (A)102 → PARTIALLY_SHIPPED5 → PARTIALLY_INVOICED
line 2 (B)104 → PARTIALLY_SHIPPED0 → NOT_INVOICED
 
ship axis    : meet = PARTIALLY_SHIPPED, join = PARTIALLY_SHIPPED  →  header.status      = PARTIALLY_SHIPPED
invoice axis : meet = NOT_INVOICED,      join = PARTIALLY_INVOICED →  header.invoiceFlag = PARTIALLY_INVOICED

Two independent componentwise folds; one status column can't carry both — which is exactly why ship and invoice are orthogonal regions, not one chain. The cross-line bundling (2×A + 4×B in one shipment) is invisible to the fold: each line accumulates only its own Σ, so the result is identical whether the goods arrived in one shipment or several. Boundary: a return would decrease shippedQty — moving an axis backwards and breaking the monotonicity the lattice assumes. Reversals need signed deltas + a net projection, not a status chain (out of scope today).
