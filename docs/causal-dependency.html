Causal dependencies between aggregates

Which aggregate's existence or state causes another's — and the data field that records it.

Verified against config/postgresql/northwood_erp.sql + the *-events / domain/ classes, 2026-06-11. This file is the single source of record for the topic (text + rendered graphs).

Northwood is event-driven and schema-per-service: aggregates never share a transaction, and there are no physical foreign keys across services (the load-bearing invariant — docs/architecture.md). A causal dependency is when one aggregate's existence or state is the cause of another's: aggregate B was created or mutated as a downstream consequence of aggregate A.

Because the flow is asynchronous (outbox → Kafka → inbox), the only durable record that "B was caused by A" is a data field on B that holds A's id. If that field is missing, the dependency lives only in event history and logs — it cannot be reconstructed from the database, so audit, reconciliation, and "why does this row exist / what produced it" can't be answered with a query.

The rule this file registers and enforces: every business-meaningful causal dependency must be explicitly tracked by a durable foreign-reference field on the caused aggregate. Cross-service links are plain UUIDs (no FK); within a schema they are real FKs — either way the relationship must be explicit and queryable (the codebase deliberately prefers a typed <thing>_id column over a polymorphic source_type/source_id pair — see the comment on inventory.stock_reservation_header).

Sagas (SalesOrderFulfilmentSaga, WorkOrderSaga, PurchaseToPaySaga) are process managers, not aggregates, and are excluded except where they carry the linking key. Reference/master data not event-sourced as aggregates: Warehouse, UnitOfMeasure, WorkCenter, GlAccount, TaxCode.

Tracking-field register

Each row is a causal edge A → B, the driving event(s) that fire it (the Kafka domain event whose inbox handler creates/mutates B — or command/in-tx where no event is involved), and the FK field on B that records the link. This is the index for "which events realise which causal dependency, and where is it stored". Verdict: typed = a dedicated typed <a>_id column (the preferred form); xor = mutually-exclusive typed columns under a CHECK; transitive = not direct, but reconstructable through one documented hop; polymorphic = a single untyped id + discriminator; MISSING = no durable field — the link is only in event history.

Verdict

The codebase is fully explicit for every load-bearing edge: every business-meaningful causal edge is recorded by a typed/xor field on the caused aggregate, with three acceptable relaxations and no open gaps — gap A resolved (the new customer_invoice_header.shipment_header_id) and gap B decided as intentional/won't-do (see below).

Acceptable relaxations (no action):

  • Transitive links (GoodsReceipt→ReplenishmentRequest via PO; Shipment→StockReservation via SO; Payment→Order via invoice) are reconstructable through exactly one documented hop and each has a supporting index. Direct duplication would be redundant.
  • JournalEntry's polymorphic source (source_document_id + source_module + source_document_type) is the one place the typed-column preference is deliberately not followed — a GL references ~11 heterogeneous document types, where 11 typed columns would be worse than the discriminated triple. Queryable via the discriminator; left as-is.

Gaps — causal dependencies not explicitly tracked

Gap A (RESOLVED) — CustomerInvoice now records the Shipment it was raised from

Resolved 2026-06-11. A commercial / balance CustomerInvoice is created from sales.SalesOrderShipped, one invoice per shipment. The event payload carries shipmentHeaderId; a nullable finance.customer_invoice_header.shipment_header_id was added (cross-context UUID, no FK, indexed) populated from it in CustomerInvoiceService.createFromShippedOrder and carried on the CustomerInvoice aggregate + CustomerInvoiceCreated payload. Null for prepayment/deposit invoices (created at placement before any shipment). So for a partially-shipped on-shipment order (several shipments → several per-shipment invoices for one SO) the (shipment, invoice, payment) fulfilment triple docs/composed-state-machines.html builds its invoice/pay rollup on is now reconstructable from the database, not only from event history.

Gap B (DECIDED — intentional, won't-do) — ReplenishmentRequest(work_order_shortage) does not record the triggering WorkOrder

When a work order's raw materials are short, inventory raises a ReplenishmentRequest with reason = 'work_order_shortage', but no source_work_order_id is stored (contrast source_sales_order_header_id/_line_id, which are stored for the sales-driven reasons). Decision (2026-06-11): leave it untracked — by design. Three reasons make a stored back-reference wrong here, not merely optional:
  1. It would be a peg that isn't one. A WO-shortage replenishment tops up the shared pool for the component (keyed product_id + warehouse_id); the WO is later un-parked by GoodsReceived matching on product_id, not by a WO-pegged fulfilment. A source_work_order_id column would read as a peg and invite WO-pegged consumption that the design deliberately avoids — unlike the sales path, where the back-reference is load-bearing (the ReplenishmentFulfilled/Cancelled fan-in routes to the saga by sourceSalesOrderHeaderId). Nothing reads a WO back-reference; it would be write-only audit decoration.
  2. It would be misleading, not just redundant. The bridge raises requests via raiseIfNoneOpen — one open request per (product, warehouse). So a second work order short on the same component while a top-up is already open is a no-op: one pool top-up serves several work orders. A single source_work_order_id could record only the first trigger — a partial, misleading provenance, not a faithful audit trail.
  3. The provenance already exists in the event log. manufacturing.RawMaterialShortageDetected carries workOrderId (+ salesOrderHeaderId/_lineId) and is keyed by it — so "which work order(s) caused top-ups for this component?" is answerable from event history without a column that would imply pool stock has WO identity.
  4. It has no bearing on event ordering — including a future root-id re-keying. The concern worth ruling out: does leaving this gap risk event disordering, especially if we ever re-keyed events by a process root id (per Event causal dependencies & the record-key question below)? No — on two counts. First, the column is a stored DB field, never a Kafka partition key: keys come from envelope.aggregateId() in KafkaEventPublisher, so the column's presence or absence cannot order or disorder any event. Second, re-keying the supply chain by workOrderId would require workOrderId on the events, and it deliberately isn't there: inventory.ReplenishmentRequested carries productId / warehouseId / reason / sourceSalesOrderHeaderId but no work-order field, and everything downstream (PurchaseRequisitionCreated → PurchaseOrderCreated → GoodsReceived) is rooted at purchaseOrderId. This WO→replenishment hop is one of the deliberate cross-process decoupling points the "no single global root key exists" conclusion below is built on: because raiseIfNoneOpen makes one pool top-up serve several WOs, there is no single workOrderId that could key it without corrupting the others' ordering. The missing column is therefore a symptom of that decoupling, not a cause of any ordering risk — and the WO-shortage flow doesn't depend on Kafka order anyway: the parked WO is un-parked by atomic GoodsReceived product-match + the saga's order-tolerance (source-state guards, advisory-lock dedup, optimistic version), not by delivery order.
This keeps the register's MISSING verdict, now explicitly justified per the Maintenance rule: the link is deliberately pool-not-peg, its provenance lives in the driving event, and it is irrelevant to partition-key ordering — so a durable column would add a misleading half-truth, not a missing guarantee.

Event causal dependencies & the record-key question

The register above is the aggregate causal graph. Lifting it to events gives the question that actually matters for Kafka: which events must be processed in order, and what record (partition) key guarantees it?

Lifting aggregate → event causal dependency. For each aggregate edge A —(E)→ B: the event E that creates/mutates B is causally after whatever produced A, and every event E_B that B emits is causally after E (B can't emit before it exists). So the event causal DAG is the aggregate causal DAG. The order-to-cash spine, read as events:

SalesOrderPlaced ≺ StockReservationRequested ≺ StockReserved ≺ ShipmentPosted ≺ SalesOrderShipped ≺ CustomerInvoiceCreated ≺ CustomerPaymentReceived

The key principle. Kafka orders two events only if they carry the same partition key. Today the key is envelope.aggregateId() = the emitting aggregate's PK (KafkaEventPublisher). That guarantees ordering only within one aggregate's own stream, not along a cross-aggregate causal chain — so the chain above is not Kafka-ordered (each event is keyed by a different id), and the saga relies on order-tolerance instead (FOR UPDATE SKIP LOCKED + optimistic version + advisory-lock dedup + source-state guards — see docs/messaging.md).

Theory: can a single record key cover all causally-dependent events?

Tested: one id (ideally the root aggregate instance id) as the key for every event in a causal dependency. Verdict: it works per independent process, but NO single global key exists — and the conflict is fundamental, by design.

The causal graph is not a forest of disjoint single-root trees; it is a DAG with deliberate decoupling and fan-in. The conflicts, verified in code:

  1. Make-to-stock pooling severs the sales↔supply root (the decisive one). Supply is triggered through inventory's shared pool (ReplenishmentRequest), not pulled by one order. So for any reorder-/pool-driven flow the supply events carry no sales order: WorkOrderManufacturingCompleted.salesOrderHeaderId and ReplenishmentFulfilled.sourceSalesOrderHeaderId are nullable (null for pool), and GoodsReceived has no SO field at all. → no salesOrderId can key the supply chain. This is the intentional inventory-as-MRP-orchestrator design (docs/sagas.md), not an oversight.
  2. Fan-in / batching aggregates have their own root. A PurchaseOrder batches requisitions; a GoodsReceipt is rooted at its PO; a Payment is rooted at paymentId and can allocate across invoices. These belong to the P2P root (purchaseOrderId), independent of any sales order.
  3. Shared resources have no upstream root. stock_balance is per (product, warehouse), touched by every order and work order; JournalEntry is fed by all chains. No single upstream id roots them (stock writes are made safe by atomicity, not ordering; GL postings have no inter-posting order).

What does hold: the graph cleanly decomposes into a handful of independent processes, each with a real single root that would totally-order that process if used as the key:

Order-to-cashsalesOrderIdevery O2C event already carries it (re-keyable today, no schema change)

The cross-process links — ReplenishmentRequested/Fulfilled (sales↔supply), GL postings, stock writes — are exactly the points the architecture deliberately makes order-tolerant (pool decoupling + atomic stock SQL + saga concurrency control). They neither have a common root nor need ordering, which is precisely why the current emitting-aggregate key is sufficient despite not enforcing the chains.

Conclusion for the theory. Root-aggregate-id as a universal key is not achievable (the make-to-stock decoupling and shared/batched resources guarantee multiple roots). But the weaker, useful form does hold: within each independent process the root correlation id is a valid single key, and for order-to-cash specifically that key (salesOrderId) is already present on every event — so if you ever wanted the O2C saga's cross-aggregate orderings enforced by partitioning instead of by order-tolerant handlers, re-keying the O2C events from emitter-PK to salesOrderId is a viable, conflict-free change (trade-off: loses per-emitter-aggregate ordering, and concentrates one order's events on a single partition → hot-partition risk for very large orders).

Maintenance

When a new aggregate or a new event-driven creation/mutation path is added, add its row to the tracking-field register and confirm the caused aggregate carries a typed field for its cause. A new edge with a MISSING verdict is a code-review fail unless it is explicitly justified here (transitive or deliberately-polymorphic). When adding a new event, record its driving-event row and, if it joins a causal chain, confirm whether its key/correlation id matches that chain's root (per Event causal dependencies & the record-key question).
