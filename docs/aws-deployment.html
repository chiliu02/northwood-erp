The three tiers (stacked, 1 EC2 each)

  • Public subnet — 1 EC2 (public IP), web-sg opens :80 + :8090 + :8089 + :8080: a guest front door (nginx static page on :80), the operational ERP SPA (nginx on :8090 serving erp-web-ui), erp-web-ui-bff (:8089), and Keycloak (:8080). The only internet-facing box; Keycloak is here so the browser can reach it for OIDC. The guest front door is the first thing to open, and its canonical copy is hosted off the fleet on an S3 static website (the friendly DNS name aliases to it), so it stays reachable even when the fleet is stopped — its built-in "Demo hours" notice then explains the app is asleep until the next weekday start. Its "Enter the ERP" link is baked from ERP_URL (the SPA on :8090) — dead while the fleet is stopped, live once it's up. The web box still serves an identical copy on :80 when running (rendered the same way from config/welcome.html.template) — this is the IP / no-custom-domain fallback (when front_door_domain is empty there is no S3 site, and the bare-IP :80 page is the only front door, via terraform output front_door_url). With a domain set, only the DNS target lives on S3. The SPA nginx reverse-proxies /api, /oauth2, /login, /logout to the BFF so the SPA's relative calls + the OIDC code flow run same-origin (no CORS); the BFF uses server.forward-headers-strategy=framework so it builds the OIDC redirect_uri against the public :8090 origin.
  • Private subnet #1 (app) — 1 EC2 (no public IP), app-sg allows only the web tier: the 7 services.
  • Private subnet #2 (infra) — 1 EC2 (no public IP), infra-sg allows only the app tier: Postgres + Kafka (+ optional observability). Most-protected, two hops from the internet.

Egress — a NAT instance (why, and the trade-off)

  • Why a NAT instance, not VPC endpoints or a NAT Gateway. The two private subnets need outbound internet to (a) pull the third-party images (Postgres/Kafka from Docker Hub, Keycloak from quay.io) and (b) let the services reach the public Keycloak issuer for JWKS. A managed NAT Gateway would do it but costs ~$32/mo+; the pure-VPC-endpoints route avoids NAT but only covers ECR — Docker Hub/quay images would then need an ECR pull-through cache (with credentials) or pre-baking, plus a Keycloak issuer/JWKS split. For a throwaway demo that's a lot of moving parts, so we use a small NAT instance: a t3.nano EC2 in the public subnet doing iptables masquerade (source_dest_check off), with the private route tables defaulting to it.
  • What it buys: private boxes pull images straight from any registry, and services reach the public Keycloak issuer through it — so no ECR image-mirroring and no Keycloak issuer/JWKS split (services just set KEYCLOAK_ISSUER_URI to the public host). The free S3 gateway endpoint still keeps S3 + ECR image-layer traffic off the NAT.
  • Trade-off: a NAT instance is a single, unmanaged box — fine for a demo, not HA. For production use a managed NAT Gateway (per AZ) — see aws-architecture.html.

Accessing Grafana

  • Grafana runs on the infra EC2 (:3000) with no public IP, and infra-sg never opens :3000 — to the internet or even tier-to-tier — so it is not browser-reachable directly. Reach it over an SSM port-forward (no SSH, no public endpoint, nothing to open): 
    terraform output grafana_access_hint
# aws ssm start-session --target <data-instance-id> \
#   --document-name AWS-StartPortForwardingSession \
#   --parameters portNumber=3000,localPortNumber=3000
# then open http://localhost:3000
  • Only present when enable_observability = true. The tunnelled Grafana runs anonymous-admin (login form disabled) — safe precisely because it is never internet-exposed; the SSM tunnel is the access control. Exposing it publicly would mean hardening that auth first.

What's gone

  • ALB, NAT Gateway, ECS/Fargate, Service Connect, Secrets Manager, CloudWatch — replaced by 3 EC2s running docker-compose, passwords/configs staged in the encrypted S3 bucket and pulled by user-data, logs to Loki. SSM-only shell (no SSH/bastion).
  • Trade-offs: no HTTPS and no HA (single instance per tier, HTTP on the public IP) — a stable address (Elastic IP / Route 53) is worth adding for the OIDC redirect URI.

Pause / resume (cost saver)

  • Stop, don't destroy, to pause. terraform destroy tears the demo down; to just stop paying overnight or on weekends, stop the EC2 fleet (web + app + data + NAT) instead. Stopping keeps the ECR images, the gp3 root volumes (so Postgres + Kafka data survive), the pinned private IPs and the web box's Elastic IP — so the Keycloak issuer URL stays valid and the resume needs no DB re-seed (every container runs --restart unless-stopped on a docker systemd service, so it returns on boot). The guest front door stays reachable while stopped — it's the S3 static site, independent of the fleet, and its "Demo hours" notice explains the app is asleep. While stopped you pay only for EBS + the Elastic IP (~$10/mo); ECR is ~$0.10/GB-mo, so keeping the images is a time saver (no rebuild/push), not a cost. It's out-of-band from Terraform — aws_instance has no power-state, so terraform plan shows no drift.
  • By hand — the two scripts discover the four boxes by the northwood-demo-* Name tag (so the NAT box, absent from the instance_ids output, is paused too): 
    ./terraform/ops/stop.ps1            # stop the fleet
./terraform/ops/start.ps1 -Wait     # start it again, block until running
  • On a weekday schedule — schedule.tf (enable_scheduler = true, the default) creates two EventBridge Scheduler schedules that call the EC2 API directly (no Lambda) to start/stop the same four boxes: start 08:30, stop 17:00, Mon–Fri, Australia/Sydney (DST-aware). MON-FRI means the fleet stays stopped all weekend — it stops Friday 17:00 and starts again Monday 08:30. Tune via start_cron / stop_cron / scheduler_timezone, or set enable_scheduler = false to remove the schedules (instances untouched). The schedules assume a least-privilege role scoped to ec2:Start/StopInstances on northwood-demo-* instances. 
    terraform apply                 # create/update the schedules
terraform output schedule       # show the active window
  • Note: a schedule only acts at the cron times — applying it mid-window won't stop a running fleet immediately (use ops/stop.ps1 for that), and the first auto-start is the next weekday 08:30.
  • AMIs are pinned so a routine apply doesn't resurrect a paused box. All four instances default their AMI to the latest Amazon Linux 2023 image (the SSM parameter /aws/service/ami-amazon-linux-latest/al2023-…); that value changes whenever AWS publishes a new image, and an ami change force-replaces the instance — recreating a running box even when the fleet was stopped, and on the data box destroying its root volume (a Postgres + Kafka data loss + re-seed). So each aws_instance carries lifecycle { ignore_changes = [ami] }: the latest-AMI lookup still picks the image at first create, but later AMI drift is ignored and applies stay non-destructive. Move to a newer AMI deliberately — set var.ami_id / var.nat_ami_id, or -replace the instance — and always terraform plan before applying when the data box matters.
