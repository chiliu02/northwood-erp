The three tiers (stacked, 1 EC2 each)

  • Public subnet — 1 EC2 (public IP), web-sg opens :8089 + :8080: erp-web-ui-bff + Keycloak. The only internet-facing box; Keycloak is here so the browser can reach it for OIDC.
  • Private subnet #1 (app) — 1 EC2 (no public IP), app-sg allows only the web tier: the 7 services.
  • Private subnet #2 (infra) — 1 EC2 (no public IP), infra-sg allows only the app tier: Postgres + Kafka (+ optional observability). Most-protected, two hops from the internet.

Egress — a NAT instance (why, and the trade-off)

  • Why a NAT instance, not VPC endpoints or a NAT Gateway. The two private subnets need outbound internet to (a) pull the third-party images (Postgres/Kafka from Docker Hub, Keycloak from quay.io) and (b) let the services reach the public Keycloak issuer for JWKS. A managed NAT Gateway would do it but costs ~$32/mo+; the pure-VPC-endpoints route avoids NAT but only covers ECR — Docker Hub/quay images would then need an ECR pull-through cache (with credentials) or pre-baking, plus a Keycloak issuer/JWKS split. For a throwaway demo that's a lot of moving parts, so we use a small NAT instance: a t3.nano EC2 in the public subnet doing iptables masquerade (source_dest_check off), with the private route tables defaulting to it.
  • What it buys: private boxes pull images straight from any registry, and services reach the public Keycloak issuer through it — so no ECR image-mirroring and no Keycloak issuer/JWKS split (services just set KEYCLOAK_ISSUER_URI to the public host). The free S3 gateway endpoint still keeps S3 + ECR image-layer traffic off the NAT.
  • Trade-off: a NAT instance is a single, unmanaged box — fine for a demo, not HA. For production use a managed NAT Gateway (per AZ) — see aws-architecture.html.

What's gone

  • ALB, NAT Gateway, ECS/Fargate, Service Connect, demo-web-ui-bff, Secrets Manager, CloudWatch — replaced by 3 EC2s running docker-compose, passwords/configs staged in the encrypted S3 bucket and pulled by user-data, logs to Loki. SSM-only shell (no SSH/bastion).
  • Trade-offs: no HTTPS and no HA (single instance per tier, HTTP on the public IP) — a stable address (Elastic IP / Route 53) is worth adding for the OIDC redirect URI.
