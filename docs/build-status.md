All seven services have empty Liquibase changelogs (V3 baseline) and an Application class. `partial` means the service has working code in production paths but doesn't cover every aggregate or status it ultimately should. Specifics:

- **sales** — `SalesOrder` aggregate, `placeOrder` / `cancel` use cases, fulfilment saga through `ready_to_ship`, compensation flow through `compensated`. Cancel rejects past `goods_shipped` (409).
- **inventory** — `StockItem` projection, `StockReservation` aggregate (sales + manufacturing-driven, with retry-cancel-then-recreate so a make-to-order saga in `raw_material_shortage` can re-emit reservation without tripping schema UNIQUE), `GoodsReceipt` + `Shipment` aggregates (post-only). `WorkOrderManufacturingCompletedHandler`: top-level WOs bump FG `stock_balance.on_hand_quantity`; sub-assembly children bump `inventory.wip_balance`. `SubAssembliesConsumedHandler` decrements WIP when a parent consumes its children. `StockMovementWriter` writes immutable audit rows from each on-hand mutation site (receipt, shipment, top-level WO completion). No `StockBalance`/`StockMovement` aggregates yet — `JdbcTemplate` directly. `wip_balance.average_cost` is 0 (parked; needs costing decision).
- **manufacturing** — `WorkOrder` aggregate (header + materials + operations) with release, operation-completion, parent-on-children gating, cancel, skipOperation. `Routing` (read, via `RoutingQueryPort`), `BomLookup`, `BomCycleDetector` (recursive-CTE walk), `WorkOrderReleaseService` (sub-assembly recursion), `WorkOrderOperationService` (cascades up parent chain on child completion), `WorkOrderPrioritisationService` (`POST /api/work-orders/{id}/priority` — pure CQRS read-side: validates WO + priority value, writes `WorkOrderPriorityChanged` straight to outbox; no aggregate state), `BomService` (cycle-detector runs on every mutation that could close a cycle), `ProductReplenishmentProjectionService` (read-side make-vs-buy classification fed by `product.MakeVsBuyChanged` + `product.ProductCreated` for type-derived defaults). Make-to-order saga through `completed`. Emits `RawMaterialShortageDetected` driving purchasing's auto-requisition, `SubAssembliesConsumed` driving inventory's WIP decrement. No BOM-create-draft endpoint (drafts via SQL/fixtures).
- **purchasing** — `Supplier` (read), `PurchaseRequisition` aggregate (auto-approves at creation, `markConverted()` for PR→PO), `PurchaseOrder` aggregate (draft / approve flow). Two creation paths: `createManual` REST (lands at draft) + `createForWorkOrderShortage` (auto-approves by config). `PurchaseOrderService.convertFromRequisition` chained from both PR-creation paths in the same txn. Inbox handlers for `RawMaterialShortageDetected`, `GoodsReceived` (matches receipt to PO line, advances P2P saga), `SupplierInvoiceApproved`, `SupplierPaymentMade`. **Supplier price list:** `purchasing.supplier_product_price` is data of record, keyed `(supplier_id, product_id, currency_code, effective_from, min_quantity)`. Authoring via `SupplierProductPriceService.setPrice(...)` (`POST /api/supplier-product-prices`) emits `SupplierProductPriceChanged` on change; no-op suppression uses `BigDecimal.compareTo` so duplicate setPrice is silent. No payment / GL posting in purchasing — finance handles GL.
- **finance** — `SupplierInvoice` aggregate (post-only) with `recordInvoice(...)` running quantity + price-variance 3-way match against `purchase_order_line_facts` (price tolerance via `northwood.finance.match.priceTolerancePercent`, default 2.0%). `CustomerInvoice` aggregate auto-invoked by `SalesOrderShippedHandler`. `Payment` aggregate (header + allocations as VO) supports both directions; DB trigger `maintain_allocation_totals` does the bookkeeping. **Perpetual-inventory GL posting** — `JournalEntryService` posts six balanced pairs as side effects: goods receipt (Dr Inventory / Cr GRNI), supplier invoice (Dr GRNI / Cr AP), supplier payment (Dr AP / Cr Bank), shipment (Dr COGS / Cr Inventory), customer invoice (Dr AR / Cr Revenue), customer payment (Dr Bank / Cr AR). Per-product-class accounts: `1210` Raw Materials Inventory, `1220` FG Inventory, `5200` Materials COGS, `5000` COGS — selected via `ProductCardLookup.findValuationClass` (backed by `finance.product_card`, fed by `product.ProductCreated` + `product.ValuationClassChanged`); missing projection rows fall back to `1200`/`5000`. `ProductCardLookup.findStandardCost` returns the projected `standard_cost` (fed by `product.StandardCostChanged`) and is consumed by `ShipmentPostedCogsHandler` for COGS — the shipment-line-stamped `unitCost` is only the documented cold-start fallback. `product_card` consolidates the former `product_standard_cost` + `product_valuation_class` (shipped §1F.6b, 2026-05-15). `LineCost(productId, amount)` records drive multi-debit (and multi-debit/multi-credit) journal helpers so each receipt/shipment posts one balanced journal (not N journals; preserves 1:1 source-document → journal mapping for reversal). `CurrencyConverter` port + `JdbcCurrencyConverter` (same-currency pass-through, inverse-rate fallback, no triangulation). REST: `POST /api/supplier-invoices`, `POST /api/payments` + `/multi` + `/customer` + `/customer/multi`, `GET/POST /api/journal-entries/{id}`, `POST /api/journal-entries/{id}/reverse`, `POST /api/journal-entries/reverse-by-source` (bulk reversal by `(sourceDocumentType, sourceDocumentId)`), `GET /api/exchange-rate?from=&to=&date=` (ad-hoc rate lookup; 404 if no rate on file). Three-step persist (header `draft` → lines → header `posted`) because of `guard_journal_line_immutability`; the `enforce_journal_balance` deferred trigger checks debit=credit at COMMIT. Manual-review queue at `GET /api/supplier-invoices/pending-review` + `POST /{id}/manual-approve` + `POST /{id}/reject`. No credit notes / refunds yet (low priority); GST tax accounts not separated from COGS/Revenue (low priority).
- **reporting** — inbox-only. Six projections wired via handlers extending `AbstractProjectionHandler` (dedupe + deserialise + recordProcessed boilerplate). `handle()` is intentionally **not** `final` — CGLIB needs to override it to apply the `@Transactional` advice; an Objenesis-instantiated proxy has uninitialised fields and NPEs on the first call if the method is final. When the same event drives two projections, each handler has its own consumer_name and lives in its own class. Sub-package handlers (`inbox/shortage/`, `inbox/atp/`, `inbox/dashboard/`) carry explicit `@Component("xxx_HandlerName")` bean names because Spring's default class-simple-name bean naming would collide. **Order-tolerance:** every projection upserts via `INSERT ... ON CONFLICT DO UPDATE` — a lifecycle event arriving before the seed creates a stub row (`(pending)` placeholders) that the late seed backfills without regressing already-advanced status or zeroing accumulators. Watch for the column-twice-in-SQL trap with string-templated `column = ?` upsert helpers — Postgres rejects with "specified more than once". Read endpoints: `/api/sales-orders/{id}/360`, `/api/purchase-orders/{id}/tracking`, `/api/work-orders/{id}/board`, `/api/material-shortages`, `/api/atp`, `/api/financial-dashboard`. Bus subscribes to `sales.events,inventory.events,manufacturing.events,finance.events,purchasing.events,product.events`. Financial dashboard snapshot endpoint (`GET /api/financial-dashboard/snapshot`, shipped 2026-05-12) populates `inventory_value` (via a `reporting.product_card` projection × ATP join), `accounts_receivable` (SUM-window over SO360), and `accounts_payable` (SUM-window over PO tracking). Only `wip_value` is still 0 — gated on a costing decision (LIFO / FIFO / weighted-avg) for `wip_balance.average_cost`.

When fleshing out a non-product service, mirror the product-service package layout and conventions before improvising.
